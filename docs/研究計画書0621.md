## 1. Research question

中心的な問い（format 非依存）：

> personalization / memory benchmark におけるモデルの正答は、本当にユーザー履歴中の evidence に
> 依存しているのか。

副次の問い：

> この evidence-use audit を、answer format（MCQ / open-ended QA）をまたいでどう設計でき、
> 各 format で何が auditability の binding constraint になるか。

**主張の中心（一文）：**
> 本研究は、パーソナライゼーション評価における correctness を evidence use と同一視する危険性を、
> answer format をまたいで監査する framework として提示する。PersonaMem では MCQ における
> default-solvability を主実証とし、LongMemEval では open-ended における sufficiency と contingency の
> judge-mediated audit を行う。ただし contingency は positive result ではなく、format-specific
> binding constraint を明らかにする threat analysis として扱う。

---

## 2. Contributions

- **C1 — Format-invariant evidence-use audit framework**: benchmark correctness と
  evidence-contingent response selection を区別する後付け監査を、L1 default-solvability /
  L2 evidence sufficiency / L3 evidence contingency の3層として **format 非依存に定義**する。
  各 layer の binding constraint が answer format で交代することを示す（§4）。固定語彙:
  hollow correctness = contingency/reversal 限定 / sufficiency ≠ necessity。
- **C2 — MCQ における default-solvability の主実証（PersonaMem）**: judge 独立・model-free・
  QC pass の confirmatory 主実証。default-solvability が benchmark 内で type 依存であり、
  別系列強モデルでも大半が残存することを示す（§5）。
- **C3 — open-ended への audit logic の拡張と format-specific constraint（LongMemEval）**:
  同一 audit logic を open-ended QA に適用したとき、target determinacy / judgeability が
  binding constraint になることを、determinacy filter と judge validation gate を経て示す。
  sufficiency（2a）は confirmatory 候補、contingency（2b）は threat analysis（§6）。

## 3. Background / Landscape

### 3.1 問題設定

LLM の personalization / memory 評価では、応答が正解に一致しても、それが (a) gold answer を
支える evidence に追従した結果か、(b) 安定特性・人気事前分布・表層一致・長文脈からの superficial
retrieval（**hollow correctness**）で偶然一致しただけかが区別されない。単一条件の正答だけでは
(a) と (b) を弁別できない。本研究は、correctness を default-solvability / sufficiency / contingency に
分解して監査する。

本研究は **特定ベンチマークの欠陥指摘を主目的としない**。既存 benchmark の correctness が
gold-supporting evidence への contingent selection をどの程度要求しうるかを、後付けで監査できるか
自体を診断する。既存ベンチは第三者が gold を定義しているため circularity を回避できる。

### 3.2 既存 ablation / counterfactual 評価との差分

Contrast Sets / CheckList DIR の発想を personalized response の evidence axis に適用。差分は、
(i) 操作対象を gold-supporting evidence に限定、(ii) counterfactual を作るのに必要な情報を既存ベンチが
持つか自体を測る、(iii) 既存ベンチへの後付け監査。

### 3.3 Landscape（既存 personalization benchmark の地図）

判定方法 3類型: label-match（judge 不要）/ ref-metric（ROUGE 等）/ LLM-judge。
記号: ● 実走確認 / ○ 構造的に適用可能・未実走 / △ 型依存 / ✗ 不可。

| Benchmark | 形式 | 判定方法 | L1 | L2 | L3 |
|---|---|---|---|---|---|
| PersonaMem v1 | MCQ | label-match | ● | △（pref のみ traceable）| △（subset のみ）|
| HorizonBench | MCQ | label-match | ○ | ○ | ○ |
| LongMemEval | open | LLM-judge | ● | ●（KU feasibility subset）| ✗/future（KU は recency 隔離）|
| LoCoMo | open QA | LLM-judge | ○ | ○ | ✗/future |
| PERMA | MCQ + interactive | label-match / judge | ○ | ○ | △（MCQ 部分）|
| PersonaMem-v2 | MCQ + open | label-match / judge | ○ | ○ | partial |

landscape から確定する含意：
1. **forced-choice は少数派**（純 MCQ は PersonaMem-v1 と HorizonBench）→ MCQ 限定では
   「personalization benchmarks を監査する」と言えない。**open-ended を扱う動機**。
2. **judge 非依存は label-match 型に限った特性** → open-ended では judge-mediated が不可避（§6）。
3. **L1 は全形式に乗る**（実走 ● 2点）。
4. PersonaMem を主ケースにするのは、traceable な preference subset と design 分析に有用な
   non-eligible 型を併せ持つため。HorizonBench は構造的に clean な将来の対象（本実証に入れない・§7）。

> 注: landscape の評価形式 / 判定方法は論文記載の一次確認。三層 applicability は本研究プロトコルからの
> 構造的判定（二次）。要 citation 最終確認の項は本文 TBD で管理。

## 4. Framework（format-invariant 定義）

先生要求（「MCQ でも自由生成でも使える枠組みに」）の核は **layer 定義の format 非依存性**で
あって、両 format で対称な behavioral 実証ではない。framework は format-invariant に定義し、
各 layer の binding constraint が format で変わることを示す。

### 4.1 3層（定義は format 非依存）

| Layer | 定義（format 非依存） | 0619e 対応 |
|---|---|---|
| L1: Default-solvability | 履歴・evidence なし（question / +options）で解けるか。correctness の shortcut/prior 成分を screen | Screen |
| L2: Evidence sufficiency | 指定 evidence だけで解けるか。item が evidence で support され得るか | Span |
| L3: Evidence contingency | evidence 内容を変えると応答も変わるか。evidence use を最も強く示し得るが confound 最多 | Remove / Change |

### 4.2 Binding constraint の format 依存性（C1 の核）

| Layer | MCQ（PersonaMem）の binding constraint | open-ended（LongMemEval）の binding constraint |
|---|---|---|
| L1 | option artifact / leakage。**judge 不要**で測れる | abstention・contamination 統制 |
| L2 | answer-space closure / option validity。**judge 独立**に測れる | **target determinacy / judgeability**（judge-mediated）|
| L3 | label fixation（letter 固着が evidence 非感受性と交絡）| target determinacy / judge 安定性 + snippet-presence への非特異反応 |

**framework の一般性は、この表（定義の一般性 + constraint のシフト）で示す。両 format で同じ深さの
実証では示さない**（実証の depth は非対称：§5, §6）。

## 6. Study 2: LongMemEval open-ended — 層別 audit

**一枚岩にしない。** sufficiency（2a）= confirmatory 候補 / contingency（2b）= threat analysis。
一律 main にも一律 feasibility にもしない。

### 6.1 determinacy filter（共通前処理）

- 正解 target が一意に定まる（target determinacy）/ gold evidence が特定できる（evidence localization）。
- KU item は競合する古値・新値を構造的に含み single-source localization に反するため除外（0619e 継承）。
- 通過 item のみが 2a/2b の母集団。2a の confirmatory は determinacy filter 通過が前提。

### 6.2 judge validation gate（新防御線・S1 supersede の核）

open-ended で judge を使う以上、judge の妥当性検証を本実験の**前**に通すゲートとする。

- judge prompt 固定（version 管理）/ 人手 spot-check / FP・FN 確認 / 判定不能除外基準 / target 曖昧の扱い。
- 合否（事前閾値）：raw agreement < 0.80 の type/層は behavioral claim から除外し記述に留める
  （閾値は judge prompt 確定後に校正し得る仮置き、構造は確定）。
- **ラベル分布が偏ると raw agreement は過大評価**されるため FP/FN 偏りを確認。**FP 集中層は
  閾値超でも除外**し judgeability constraint として報告（open-ended では「evidence に基づかないのに
  judge が意味的近接で正解扱い」する FP が sufficiency の confirmatory を内側から崩すため最優先）。

### 6.3 Study 2a: Evidence sufficiency（confirmatory 候補）

- 問い：十分な evidence があれば answer を復元できるか。determinacy + judge gate 通過 item に限り
  judge-mediated でも比較的 clean。
- 予備データ：LongMemEval C5（evidence_sensitive 12/20）。20 の抽出法を確認し、無作為でなければ
  feasibility observation と呼ぶ（confirmatory と過称しない）。
- 位置づけ：judge gate + determinacy filter 通過し十分な n が確保できれば **main result に含め得る層**。
  n 不足なら open-ended の sufficiency audit の **eligibility constraint** として報告。

### 6.4 Study 2b: Evidence contingency（threat analysis として設計）

- 問い：evidence を変えると answer が変わるか。**最初から threat analysis として設計**し positive を
  狙わない。
- 根拠：MCQ 側（旧 pilot）で placebo leak / default-like target return / evidence-shift target leak /
  snippet-presence bias が観測。これらは open-ended でも出る蓋然性が高く、format を超えて contingency
  が binding constraint であることを示す診断に留める。旧 pilot の guardrail を事前 guardrail に転用。
- 位置づけ：feasibility-gated threat analysis。binding constraint の format 依存性を示す。

### 6.5 旧 PersonaMem constructed-evidence pilot の扱い

旧 pilot は **最終主張を固定する confirmatory result ではなく、設計上のリスクを発見する
exploratory design stress-test** として扱う。Study 2 設計の guardrail の根拠。Study 1
（Gate1/type/M3）は confirmatory 主実証であり、旧 pilot のみが exploratory（混同しない）。

## 7. Claim boundary

- **言える：** correctness には default-solvable component が含まれ得る（accuracy を evidence use と
  同一視できない）。default-solvability は PersonaMem 内で type 依存。strict の大半は別系列強モデルでも残存。
- **言える：** framework は format-invariant に定義でき、binding constraint が format で変わる。
- **言わない：** strict item が evidence を要求する（L1 射程限界）。
- **言わない：** framework を cross-benchmark / cross-format で validate した（**定義の一般性**を示すに留め、
  実証の網羅では主張しない）。
- **言わない：** contingency の positive result（2b は threat analysis）。
- **モデル一般化：** 8B×2 + 27B×1。「強モデル一般」には一般化しない。
- HorizonBench は本実証に入れない（type 異質性が同一データで「default は一枚岩でない」を示せたため）。
  条件照合後に補助観察として可否判断。

  として報告（§6.3）。
