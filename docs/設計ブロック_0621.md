## 1. 方針（一文固定）

> 本研究は、パーソナライゼーション評価における correctness を evidence use と同一視する危険性を、
> answer format をまたいで監査する framework として再構成する。PersonaMem では MCQ における
> default-solvability を主実証とし、LongMemEval では open-ended における sufficiency と
> contingency の judge-mediated audit を行う。ただし contingency は positive result ではなく、
> format-specific binding constraint を明らかにする threat analysis として扱う。

中心的な問い（format 非依存）：

> personalization benchmark におけるモデルの正答は、本当にユーザー履歴中の evidence に
> 依存しているのか。

## 2. Framework（format-invariant 定義）

先生の要求（「MCQ でも自由生成でも使える枠組みに」）の核は **layer 定義の format 非依存性**で
あって、両 format で対称な behavioral 実証ではない。よって framework は **format-invariant に定義**し、
各 layer の **binding constraint（律速する脅威）が format で変わる**ことを示す。

### 2.1 3層（定義は format 非依存）

| Layer | 定義（format 非依存） |
|---|---|
| L1: Default-solvability | ユーザー履歴・evidence なし（question のみ／+ options）で解けるか。correctness に含まれる shortcut/prior 成分を screen する |
| L2: Evidence sufficiency | 指定された evidence だけで解けるか。item が evidence によって support され得るか |
| L3: Evidence contingency | evidence の内容を変えると応答も変わるか。evidence use を最も強く示し得るが confound が最も多い |

### 2.2 Binding constraint の format 依存性（先生要求の核）

| Layer | MCQ（PersonaMem）の binding constraint | open-ended（LongMemEval）の binding constraint |
|---|---|---|
| L1 | option artifact / leakage（gold が option に解決可・推測可）。**judge 不要**で測れる | abstention・contamination 統制（履歴なしは原理的に解けない→0%でなければ汚染検出） |
| L2 | answer-space closure / option validity。**judge 独立**に測れる | **target determinacy / judgeability**（judge-mediated）。判定が安定する item に限る |
| L3 | label fixation（letter への固着が evidence 非感受性と交絡） | target determinacy / judge 安定性 + snippet-presence への非特異反応 |

**framework の一般性は、この表（定義の一般性 + constraint のシフト）で示す。**
両 format で同じ深さの実証を行うことでは示さない（実証の depth は非対称：§3, §5）。

## 4. Judge Validation ゲート（Study 2 の前置・S1 supersede の防御線）

open-ended で judge を使う以上、**judge の妥当性検証を本実験の前に通すゲート**とする。
これを省くと「judge を使う場合に判定基準を明示する」という新しい防御線が空文化する。

### 4.1 ゲート項目

- judge prompt の固定（version 管理）
- 人手 spot-check（judge 判定と人手判定の照合）
- judge の false positive / false negative の確認
- 判定不能 item の除外基準（事前定義）
- target が曖昧な item の扱い（§5.1 の determinacy フィルタと連動）

### 4.2 合否基準（事前閾値・恣意性を排す）

- judge-人手の一致を **raw agreement（必要に応じ κ または disagreement pattern を併記）**で測る。
- **閾値（仮置き）：raw agreement < 0.80 の item type / 層は behavioral claim から除外**し、
  記述に留める。閾値の具体値は judge prompt 確定後に校正し得るが、「behavioral 実証の前に
  閾値を置く」構造自体は確定。
- **ラベル分布が極端に偏る場合、raw agreement だけでは安定性を過大評価する**ため、
  false positive / false negative の偏りを確認する（judge も人手も "correct" 偏重なら
  agreement は高く見えるが安定とは限らない）。
- **特に false positive が集中する item type / 層は、raw agreement が閾値を超えていても
  behavioral claim から除外し、judgeability constraint として報告する。** open-ended では
  「evidence に基づかないのに judge が意味的近接で正解扱いする」FP が sufficiency の
  confirmatory を内側から崩すため、FP 点検を最優先の防御線とする。
- judge が不安定な層は、その時点で behavioral 実証を諦め、constraint の所在として記述報告する
  （negative も framework の貢献に転換）。

## 5. Study 2（open-ended / LongMemEval）：層別 audit

**Study 2 を一枚岩にしない。** sufficiency（2a）と contingency（2b）で格が異なる。
一律 main にも一律 feasibility にもしない（S3）。

### 5.1 共通前処理：determinacy フィルタ

- 正解 target が一意に定まる item に限定（target determinacy）。
- gold evidence が明確に特定できる（evidence localization）。
- KU（knowledge-update）item は競合する古値・新値を構造的に含み single-source localization に
  反するため除外（既存方針を継承）。
- このフィルタを通った item のみが 2a/2b の母集団。**2a の confirmatory は determinacy フィルタ
  通過が前提**（target が曖昧な item では sufficiency すら judge 依存で揺れる）。

### 5.2 Study 2a：Evidence sufficiency（confirmatory 候補）

- 問い：十分な evidence があれば answer を復元できるか。
- determinacy フィルタ通過 + judge validation ゲート通過 item に限り、judge-mediated でも
  比較的 clean（正解が evidence で決まり judge の仕事が照合に近い）。
- 予備データ：LongMemEval C5（evidence_sensitive 12/20）。ただし 20 の抽出法を確認し、
  無作為でなければ feasibility observation と呼ぶ（confirmatory と過称しない）。
- 位置づけ：**judge validation と determinacy フィルタを通過し、十分な n が確保できた場合には
  main result に含め得る層**。n が不足する場合は、open-ended 形式における sufficiency audit の
  **eligibility constraint**（sufficiency ですら item eligibility に制約される）として報告する。

### 5.3 Study 2b：Evidence contingency（threat analysis として設計）

- 問い：evidence を変えると answer が変わるか。
- **最初から threat analysis として設計**。positive な behavioral result を狙わない。
- 根拠：MCQ 側（旧 pilot）で placebo leak / default-like target return /
  evidence-shift target leak / snippet-presence bias が観測された。これらは open-ended でも
  出る蓋然性が高く（むしろ生成形式の方が snippet 存在に反応しやすい）、format を超えて
  contingency が binding constraint であることを示す診断に留める。
- 旧 MCQ pilot の guardrail（上記4現象）を open-ended 設計の事前 guardrail として使う。
- 位置づけ：**feasibility-gated threat analysis**。binding constraint の format 依存性を示す。

### 5.4 feasibility-gated な進め方

1. evidence-use audit に適した item を選別（determinacy フィルタ）
2. judge で安定判定できるか確認（judge validation ゲート、§4）
3. 少数〜中規模で 2a（sufficiency）/ 2b（contingency）を試す
4. 結果をもとに open-ended の制約と可能性を整理

## 6. 旧 PersonaMem 実験の扱い（防御的再分類）

旧 PersonaMem / Gate 2 / behavioral pilot は、**最終主張を固定する confirmatory result では
なく、設計上のリスクを発見する exploratory design stress-test** として扱う。よって新 framework
設計に合わせて実験単位を再構成することは問題ない。これらは「なぜ contingency を最初から
threat analysis として設計すべきか」の根拠（guardrail）として価値を持つ。

※ Study 1（Gate1/type異質性/M3）は exploratory ではなく **confirmatory 主実証**（§3, FROZEN）。
旧 constructed-evidence pilot のみが exploratory stress-test。両者を混同しない。

## 7. Claim boundary（確定）

- **言える：** correctness には default-solvable component が含まれ得る（accuracy を evidence use と
  同一視できない）。default-solvability は PersonaMem 内で type 依存。strict の大半は別系列強モデルでも残存。
- **言える：** framework は format-invariant に定義でき、binding constraint が format で変わる。
- **言わない：** strict item が evidence を要求する（Gate 1 射程限界）。
- **言わない：** framework を cross-benchmark / cross-format で validate した（**定義の一般性**を示すに留め、
  実証の網羅では主張しない）。
- **言わない：** contingency の positive result（2b は threat analysis。binding constraint のシフトを
  示すに留める）。
- **モデル一般化：** 8B×2 + 27B×1。「強モデル一般」には一般化しない。

## 10. Limitation（確定）

- モデル規模・系列：8B×2 + 27B×1。strict は「これらが解けない」であり「evidence 要求」ではない。
- strict の頑健さの質は type 依存（capability-robust / capability-sensitive）。
- cluster=20：persona 単位の推論統計は載せない。type 差・persona 差は descriptive。
- Study 2b（contingency）は negative-leaning を前提に設計（threat analysis）。
- open-ended は judge-mediated：judge validation ゲートを通った層のみ behavioral claim に使う。
